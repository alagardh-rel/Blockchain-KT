What is Solana?

Solana is a high-performance blockchain network designed to support decentralized applications and cryptocurrency transactions at scale.

In Simple Terms:

Imagine a bank that's run by thousands of computers instead of one company. Anyone can join, no one owns it, and everyone sees all transactions. That's Solana.

Key Characteristics

  • Fast: Process ~400-1000+ transactions per second
  • Cheap: Transaction costs under $0.00001
  • Secure: Uses cryptographic signatures to prevent fraud
  • Decentralized: Thousands of validators maintain the network
  • Scalable: More validators = more throughput

Why Was Solana Created?

Problem: Bitcoin and Ethereum are too slow and too expensive for everyday use.

Bitcoin

  • ~7 transactions/second
  • $1-$5 per transaction
  • Takes ~10 minutes to confirm

Ethereum

  • ~15 transactions/second
  • $2-$100 per transaction
  • Takes ~15 seconds to confirm

Solana's Solution

Solana achieves 400-1000+ TPS by:

  • Using Proof of History (new innovation)
  • Parallel transaction processing
  • Optimized consensus mechanism
  • No state bloat (keeping data lean)

Key Concepts Explained

Validator (Network Participant)

Simple Definition: A powerful computer running Solana software that:
  • Receives transactions from users
  • Verifies they're correct
  • Executes them
  • Votes on the correct result
  • Gets rewarded with SOL

Node (Computer in Network)

Any computer running Solana software. Can be a validator (voting) or a regular node (just observing).

Leader (Current Block Producer)

Each round:
  • One validator is chosen as "Leader"
  • Leader bundles recent transactions
  • Creates a new block
  • Broadcasts to network
  • Gets transaction fees as reward
  • Then someone else becomes leader

Slot

A unit of time: The Solana network produces a new block approximately every 400 milliseconds. Each block produced in this time window is called a "slot".

Epoch

430,000 slots = ~5 days. At the start of each epoch, validators are reshuffled and assigned new responsibilities.

Finality

When is a transaction "done"?
  • Submitted: ~0 seconds
  • Included in block: ~400ms
  • Confirmed by validators: ~2 seconds
  • Finalized (can't be reversed): ~6.4 seconds

Bitcoin: 10 minutes | Ethereum: 15 seconds | Solana: 6.4 seconds

Proof of History (PoH)

The Problem PoH Solves

Traditional blockchains need to agree on "time". But in a decentralized network, different computers have slightly different clocks. How do they agree?

Bitcoin's Solution (Proof of Work)

  • Miners solve math puzzles
  • Solving takes ~10 minutes
  • Time = puzzle difficulty
  • Very energy inefficient

Solana's Solution (Proof of History)

Simple Idea: Create a historical record that proves an event happened at a specific moment in time.

How:

  1. Take previous hash (string of data)
  2. Add current transaction
  3. Create new hash
  4. Repeat continuously

The hash chain proves that events happened in this order at these times, without needing a central clock.

Why It's Efficient

  • No expensive math puzzles
  • Pure computation (can't be faked)
  • Minimal energy usage
  • Scales linearly

Proof of Stake (PoS)

The Voting System

After transactions are produced (via PoH), validators vote to confirm them:

  1. Stake: Validators "lock up" their SOL as collateral
  2. Vote: Validators vote on valid transactions
  3. Reward: Correct voters earn inflation + fees
  4. Penalty: Dishonest voters lose stake

Why It Works

Incentives align:
  • Validators want SOL price to go up (they own it)
  • If they cheat, they lose money
  • If they help network, they earn rewards
  • The network stays secure by design

Tower BFT (Voting Mechanism)

Tower BFT is Solana's consensus algorithm:

  • Validators vote on blocks they see
  • Each vote makes previous blocks harder to reverse
  • After 2/3 of validators vote: transaction is likely final
  • After most validators vote: completely irreversible

Solana Architecture Overview

The Pipeline: How Solana Works

1. Gulf Stream (Mempool)

  • Transactions enter here first
  • Leader selection is known in advance
  • Can forward transactions to next leaders
  • Reduces ordering delays

2. Turbine (Block Broadcasting)

  • Leader propagates block to network
  • Validators form a tree structure
  • Each validator forwards to others
  • Reduces bandwidth requirements

3. Sealevel (Execution Engine)

  • Executes transactions in parallel
  • Non-conflicting transactions can run simultaneously
  • Increases throughput dramatically
  • Uses accounts to prevent conflicts

4. Pipelined Proof of Stake (Consensus)

  • Validators vote on blocks
  • Votes accumulate continuously
  • Finality achieved in ~6.4 seconds
  • Next validators start producing new blocks

The Complete Flow

You send a transaction:
    ↓
Gulf Stream (mempool):
├─ Sits with other pending txs
├─ Leader is known in advance
└─ Gets sent to next leaders

    ↓
Leader receives block slot:
├─ Collects pending transactions
├─ Orders them
└─ Creates a block

    ↓
Turbine broadcasts:
├─ Leader sends to validators
├─ Validators relay in tree
└─ Network receives in ~400ms

    ↓
Sealevel executes:
├─ Parses transactions
├─ Executes in parallel
├─ Updates state
└─ ~200ms execution time

    ↓
Validators vote (Tower BFT):
├─ Vote on validity
├─ Votes accumulate
├─ 2/3 consensus = likely final
└─ Most votes = absolutely final

    ↓
Transaction FINAL (~6.4 seconds total)

🔐 Cryptography Fundamentals

Ed25519 Elliptic Curve

What Solana Uses:
  • Ed25519 elliptic curve cryptography
  • 256-bit keys (32 bytes)
  • 64-byte signatures
  • Deterministic (same input = same signature)
  • Fast and secure

Digital Signature Process

1. Private Key (secret):
   ├─ 32 random bytes
   ├─ NEVER share this
   └─ Kept in wallet/hardware device

2. Public Key (shareable):
   ├─ Derived from private key
   ├─ 32 bytes
   ├─ Safe to share
   └─ Proves you authorized transaction

3. Message (transaction data):
   ├─ Accounts involved
   ├─ Amount
   ├─ Instructions
   └─ Recent blockhash

4. Signing:
   ├─ Hash the message
   ├─ Encrypt hash with private key
   ├─ Get 64-byte signature
   └─ Signature proves: "I authorized this with MY private key"

5. Verification:
   ├─ Validators have your public key
   ├─ Decrypt signature with public key
   ├─ Compare with message hash
   ├─ Match = authentic transaction
   └─ No match = rejected (fraud)

Why This Works

The Math:
  • Given: signature and public key
  • Impossible to: find private key
  • Only way to sign: have private key
  • Result: Authentication without sharing secret

From Seed Phrase to Address

    ↓ (BIP44 - Derivation Path)

    ├─ Derived mathematically
    ├─ Cannot reverse to private key

├─ Readable format
├─ Example: 8e74fbE1NHhbw6X9DQ2wE6WbH7NvqvJ2...
└─ What people see

BIP44 Standard (Hierarchical Deterministic)

Part Value Meaning
m Master Start from master seed
44' BIP44 Using standard derivation
501' Solana Coin type for Solana
0' Account First account
0' Change Non-change address
0' Index First address

Result: Same seed phrase always generates same keys, across all wallets!

Wallets

A wallet is:
  • Software that stores private keys
  • Generates addresses
  • Signs transactions
  • Shows your balance
Types:
  • Hot Wallet: Private key on computer/phone (convenient, less secure)
  • Hardware Wallet: Private key on device (very secure, harder to use)
  • Web Wallet: Private key in browser (risky, easy to use)

💾 Account Model

What is an Account?

An account is a data container with:
  • Address: Unique identifier (58 chars)
  • Lamports: SOL balance (1 SOL = 10^9 lamports)
  • Owner: Program that can modify this account
  • Data: Arbitrary bytes (what owner stores)
  • Executable: Is this account a program?

Account Ownership

System Program Owned Account (Wallet):
├─ Owner: System Program (11111111111111...)
├─ Lamports: 5,000,000,000 (5 SOL)
├─ Data: empty
├─ Executable: false
└─ Only: System Program can modify

Token Program Owned Account (Token Balance):
├─ Owner: Token Program (TokenkegQfe...)
├─ Lamports: 2,039,280 (rent-exempt)
├─ Data:
│  ├─ mint: USDC address
│  ├─ owner: wallet_address
│  ├─ amount: 1,000,000,000 (1B USDC)
│  └─ ...more fields
├─ Executable: false
└─ Only: Token Program can modify

Program Account:
├─ Owner: BPF Loader
├─ Lamports: 100,000,000
├─ Data: compiled program bytecode
├─ Executable: true
└─ Only: BPF Loader can modify

Rent

Storage Cost:
  • 19.1 lamports per byte per year
  • Account (128 bytes) = 2,444 lamports/year
  • Can be rent-exempt if balance ≥ 2 years rent
  • Rent-exempt accounts stay on chain forever

Account Locking

In transactions, each account is marked:

  • Writable: Can be modified by the program
  • Read-Only: Can only be read, not modified
  • Signer: Must sign the transaction
  • Non-Signer: Doesn't need to sign

Why? Sealevel parallelizes: Non-conflicting writes can execute simultaneously!

🪙 Tokens & Token Accounts

Token Mint

Mint Account contains:
  • Supply: Total tokens issued
  • Decimals: How many decimals (USDC=6)
  • Mint Authority: Can issue more tokens
  • Freeze Authority: Can freeze accounts

Token Account (ATA)

Associated Token Account (ATA):

🔗 Transactions in Depth

Transaction Structure

Transaction {
  message: {
    header: {
      num_required_signers: 1,
      num_readonly_signers: 0,
      num_readonly_non_signers: 1
    },
    
    account_keys: [
      "alice_address",      // signer, writable
      "bob_address",        // non-signer, writable
      "system_program"      // non-signer, read-only
    ],
    
    recent_blockhash: "9q7RTfSe5VgC...",
    
    instructions: [
      {
        program_id: "11111111111111111111111111111111",
        accounts: [
          {address: "alice", signer: true, writable: true},
          {address: "bob", signer: false, writable: true}
        ],
        data: [2, 0x80, 0x84, 0x1e, ...]  // encoded instruction
      }
    ]
  },
  
  signatures: [
    "7h8a9d2k1e...w9q0p1r2s3t4u5v6w7x8y9z"  // alice's signature
  ]
}

Transaction Lifecycle

1. CREATION (User's Computer)
   ├─ Build message
   ├─ Hash the message
   └─ Sign with private key(s)

2. SUBMISSION (RPC Node)
   ├─ Send to RPC endpoint
   ├─ Node validates format
   └─ Forward to leader's mempool

3. MEMPOOL (Gulf Stream)
   ├─ Sit with pending transactions
   ├─ Wait for leader slot
   └─ Leader can see upcoming transactions

4. INCLUSION (Leader Slot)
   ├─ Leader orders transactions
   ├─ Creates block
   ├─ Add recent_blockhash
   └─ Broadcast via Turbine

5. PROPAGATION (Network)
   ├─ Validators receive block
   ├─ Parse transactions
   └─ ~400ms total

6. EXECUTION (Sealevel)
   ├─ Load accounts
   ├─ Execute program code
   ├─ Update account states
   └─ ~200ms execution

7. VOTING (Tower BFT)
   ├─ Validators vote
   ├─ Votes accumulate
   ├─ 66% consensus = likely final
   └─ 95% consensus = absolutely final

8. FINALITY (~6.4 seconds)
   ├─ Transaction irreversible
   ├─ State committed
   └─ ✓ Done

Replay Protection

Problem: Can attacker submit same tx again? Solution: Recent blockhash expires after 2-3 minutes
  • Each block has unique hash
  • Transaction includes recent_blockhash
  • If blockhash too old: transaction rejected
  • Can't replay old transactions

🛡️ Replay Protection

The Problem

Alice sends 10 SOL to Bob. A malicious node captures this transaction and broadcasts it again. Now Bob receives 20 SOL (twice!)

Solutions

1. Recent Blockhash (Most Common)

  • Transaction includes: recent_blockhash
  • Blockhash from ~100 blocks ago (~40 seconds)
  • Valid for ~2-3 minutes
  • Free to implement

2. Durable Nonce (For Delayed Signing)

  • Special account stores current nonce value
  • Transaction includes this nonce
  • After validation, nonce updates
  • Old nonce becomes invalid
  • Can submit transaction hours/days later

Replay Across Chains

New Problem: Solana and Solana Testnet are separate chains!
  • Same private key works on both
  • Could send mainnet tx to testnet
  • Testnet could replay to mainnet
Solana Solution:
  • Each chain has different bank hash
  • Transactions include chain discriminator
  • Not compatible across chains

What Validators Do

  3. Vote on block correctness

Penalties:

🔧 What Are Solana Programs?

A Solana program is smart contract code compiled to BPF bytecode, deployed on validators, stateless, invoked by transactions, and receives accounts as parameters.

In Simple Terms:

Think of a Solana program like a vending machine. You insert money (send transaction), select item (specify accounts), and get result (program executes).

Programs vs Traditional Apps

Aspect Traditional App Solana Program
Runs on One server All validators
Language JavaScript, Python Rust, C
Data storage Database Blockchain accounts
Invocation HTTP request Transaction instruction
Execution model Stateful Stateless

💾 Accounts and State Storage

The Key Difference: Programs Don't Store State

Ethereum: Contracts store data internally 
mapping(address => uint) balance;
Solana: Programs receive accounts as parameters 
fn transfer(
    from_account: &Account,
    to_account: &mut Account
) { ... }

Account Ownership Rule

ONLY the account's owner can modify its data
  • Token Account owner = Token Program
  • Only Token Program can modify balance
  • Your Program owns accounts = only you can modify

Read-Only vs Writable Accounts

Type Can Read Can Write Parallelizable
Read-Only ✓ Yes ✗ No ✓ Yes (parallel)
Writable ✓ Yes ✓ Yes ✗ No (sequential)

Why? Sealevel parallelizes: Non-conflicting writes can run simultaneously!

🔗 Cross-Program Invocation (CPI)

What is CPI?

CPI = One program calling another program from within a transaction

Real-World Example: Token Swap

Transaction: Swap 100 USDC for 500 COPE

Step 1: Call DEX Program
├─ Program: DEX Program
├─ Accounts: [alice_usdc, bob_cope, DEX program, Token Program, USDC mint]
└─ Instruction: swap(100)

Step 2: DEX Program Logic
├─ Check: Can swap 100 USDC? YES
├─ Calculate: Get 500 COPE
├─ Use CPI to call Token Program:
│  ├─ "Transfer 100 USDC from alice_usdc to dex_usdc"
│  └─ Token Program modifies accounts
├─ Use CPI to call Token Program again:
│  ├─ "Transfer 500 COPE from dex_cope to bob_cope"
│  └─ Token Program modifies accounts
└─ Return success

Result:
├─ alice_usdc: 100 → 0 tokens
├─ bob_cope: 0 → 500 tokens
├─ All atomic (can't have partial swap)
└─ Both programs' logic executed

Program Derived Signers (PDA)

Problem: Only wallets have private keys to sign Solution: PDA (Program Derived Address)
  • No private key (never will have one)
  • Only program can authorize signing
  • Deterministic (same inputs = same address)
  • Used as: program authority

🔄 Upgradeable Programs

Two Types of Programs

Immutable

  • Deployed and fixed
  • No updates ever
  • Security guarantee

Upgradeable

  • Deployed with authority
  • Can be updated
  • Flexibility + risk

How Upgradeable Programs Work

Structure:

Proxy Account (the actual "Program ID"):
├─ Owner: BPF Loader (Upgradeable)
├─ Data: Pointer to implementation
└─ Does NOT contain code

Implementation Account (actual code):
├─ Owner: BPF Loader (Upgradeable)
├─ Data: Program bytecode
└─ Contains actual logic

Upgrade Flow:
1. Deploy v1 implementation
2. Point proxy to v1
3. Users call proxy

4. Deploy v2 implementation (new account)
5. Authority updates proxy → v2
6. Users call proxy (same address!)
7. Proxy executes v2 code

Result:
├─ Program ID stays same
├─ Code can change
└─ Seamless upgrade

🪙 Mint and Authorities

What is a Mint?

A Mint is the definition of a token

Mint Account contains:
├─ supply (total issued)
├─ decimals (configurable)
├─ mint_authority (who can create tokens)
├─ freeze_authority (who can freeze)
└─ is_initialized (true)

Example (USDC):

  • supply: 43 billion USDC
  • decimals: 6
  • mint_authority: Circle (company)
  • freeze_authority: Circle
  • address: EPjFWaJy47gIdJtiEunGoMaxuS1yGqjNzSZThLgMjNj

Mint Authority

What it does:
  • Can create (mint) new tokens
  • Increases total supply
Risk: If private key leaked = unlimited inflation

Freeze Authority

What it does:
  • Can freeze (lock) token accounts
  • Frozen account cannot transfer
  • Can unfreeze later
Use cases:
  • Compliance (freeze sanctioned accounts)
  • Security (temporarily freeze if hacked)

❓ Common Confusions and FAQs

❌ "Programs store state, right?"

WRONG

Programs are stateless. State lives in separate accounts passed as parameters.

✓ "I can use the same program ID for different tokens?"

CORRECT

Yes! Token Program manages ALL tokens with one program ID.

❌ "Nonce accounts are like Ethereum nonces?"

DIFFERENT

Ethereum nonces are per-account and auto-incrementing. Solana nonces are special accounts.

☠️ "My private key was leaked. What happens?"

HACKED
  • Anyone can transfer all SOL
  • Anyone can transfer all tokens
  • Anyone can sign transactions
Solution:
  • Create new account immediately
  • Transfer remaining funds
  • Never reuse as signing account

🎯 Top 10 Things to Remember

  1. Programs are stateless
  2. State lives in separate accounts
  3. Only owner can modify accounts
  4. Fees are flat (5000 lamports)
  5. Finality is fast (~6.4 seconds)
  6. SPL tokens need ATA (separate per token)
  7. CPI (program calling program) is normal
  8. Rent is required for storage
  9. Durable nonce = delayed signing
  10. Deterministic + blockhash = replay protection
