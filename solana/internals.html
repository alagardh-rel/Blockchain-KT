🔐 Cryptography Fundamentals

Understanding Public Key Cryptography

Simple Analogy: Imagine two boxes with special locks:

  • You have a private key (secret key to your lock)
  • You give others your public key (the lock itself)
  • Anyone can lock something with your public key (send you money)
  • Only you can unlock it (spend money)

Ed25519: Solana's Signature Scheme

What It Is: An elliptic curve cryptography algorithm for signing transactions.

Feature ECDSA (Ethereum) Ed25519 (Solana)
Signature Size 64-65 bytes 64 bytes
Speed Slower verification Faster verification
Malleability Potential issues Immune
Implementation Complex Simpler

Base58 Encoding

Why Base58? Base64 uses confusing characters (0 vs O, l vs I, +, /). Base58 removes them! 
Base58 Alphabet:
123456789ABCDEFGHJKLMNPQRSTUVWXYZabcdefghijkmnopqrstuvwxyz
(No 0, O, I, l)

Why Solana Uses It:

  • Human-readable addresses
  • Less prone to copy-paste errors
  • Compact representation

🔑 Private Keys, Public Keys, and Addresses

The Three Tiers

Tier 1: PRIVATE KEY (Secret!)
├─ 32 bytes (256 bits)
├─ Must be kept SECRET
└─ Used to sign transactions

    ↓ (One-way derivation)

Tier 2: PUBLIC KEY
├─ 32 bytes derived from private key
├─ Can be shared publicly
└─ Used to verify signatures

    ↓ (In Solana = same as address)

Tier 3: ADDRESS
├─ 32 bytes
├─ Identified on blockchain
└─ Base58 encoded for display

Comparison with Ethereum

Feature Bitcoin Ethereum Solana
Private Key 32 bytes 32 bytes 32 bytes
Public Key Compressed 33B Uncompressed 64B 32 bytes
Address Derivation Hash of pubkey Hash of pubkey = Public key
Address Size 20 bytes 20 bytes 32 bytes
Why 32 bytes? Solana's public keys ARE the addresses (no hashing step). Better collision resistance + simpler architecture.

👛 Wallet Architecture

From Seed Phrase to Address: Step-by-Step

Step 1: SEED PHRASE (12-24 words)
"abandon ability able about above absent..."
    ↓ (BIP39 algorithm)

Step 2: MASTER SEED (64 bytes)
512 bits of binary data
    ↓ (PBKDF2 derivation)

Step 3: MASTER KEYPAIR
Derived from master seed
    ↓ (BIP44 derivation path)

Step 4: ACCOUNT KEYPAIR
Path: m/44'/501'/0'/0'/0
These are your signing keys
    ↓ (Ed25519 public key)

Step 5: ADDRESS (Base58)
32 bytes encoded for display
Public key = Address in Solana

HD Wallets and Derivation Paths

What is an HD Wallet? One seed phrase generates multiple keypairs.

BIP44 Standard

m / purpose' / coin_type' / account' / change / address_index

Examples:
m/44'/501'/0'/0'/0  = First account, first address
m/44'/501'/0'/0'/1  = First account, second address
m/44'/501'/1'/0'/0  = Second account, first address

Details:
├─ m = Root (master key)
├─ 44' = BIP44 standard
├─ 501' = Solana (Bitcoin=0', Ethereum=60')
├─ 0' = Account number
├─ 0 = Receiving (1 = change address)
└─ 0 = Address index
⚠️ Security Note: The `'` (hardened) means uses private key for derivation. More secure but child cannot be determined without private key.

💾 Solana's Account Model

Account Structure

Account:
├─ address (32 bytes, public key)
├─ owner (32 bytes, who controls this)
├─ lamports (balance)
├─ data (raw bytes)
├─ executable (is this a program?)
└─ rent_epoch (for rent tracking)

Key Difference from Ethereum: Ethereum account IS the entity. In Solana, account is a DATA STRUCTURE with an owner.

Lamports and Rent

1 SOL = 1,000,000,000 lamports
1 lamport = 0.000000001 SOL

Rent System

Annual Rent = (account_size) × 19.1 lamports/byte

Example:
128-byte data = 128 × 19.1 = 2443.2 lamports/year

Rent Exemption:
└─ Balance ≥ 2 years of rent = never pay rent

Who Pays? Transaction executor pays, usually the account owner or rent payer specified.

Ownership and Executability

Regular Account

  • executable = false
  • owner = State Program
  • data = User data

Executable Account (Smart Contract)

  • executable = true
  • owner = BPF Loader
  • data = Compiled program code

Associated Token Accounts (ATA)

The Problem: Ethereum has one address for all tokens. Solana needs separate accounts per token.

Solution: ATA

Derive deterministic address:
INPUT: owner + token_mint
└─ SHA256 hash (always same for same inputs)
OUTPUT: Same address every time

Benefit:
├─ Don't need to ask "what's your address?"
├─ Standard location for each token
└─ Can auto-create if missing

Program Derived Addresses (PDA)

What is a PDA? An address derived from program ID + seeds + bump. No private key!

PDA Generation:
INPUT: Program ID + seeds + bump

SHA512(
  "ProgramDerived",
  program_id,
  seed_1,
  seed_2,
  ...,
  bump
)

1. Hash until valid Ed25519 point
2. Increment bump if not
3. That's the PDA address
✓ Use Case: Program vault holding user funds. Only program can control it (no private key to steal).

📝 Transactions

Transaction Structure

Solana Transaction:
│
├─ Message
│  ├─ Header
│  │  ├─ Num required signers
│  │  ├─ Num readonly signers
│  │  └─ Num readonly non-signers
│  │
│  ├─ Account Addresses (list of all)
│  ├─ Recent Blockhash (replay protection)
│  └─ Instructions (list of operations)
│
└─ Signatures (list of signatures)

Message Structure: Account Metas

For each instruction, specify accounts:

[
  {
    address: "TokenkegQfeZyiNwAJsyFbPVwwQQftas5EMQQqJL8Aw",
    is_signer: false,      // Doesn't need to sign
    is_writable: false     // Read-only
  },
  {
    address: "9B5X4z6z2MPGEg25zZvzG2jhXwQaE...",
    is_signer: false,
    is_writable: true      // Will be modified
  },
  {
    address: "8e74fbE1NHhbw6X9DQ2wE6WbH7NvqvJ2C1VFhcQPg8J2",
    is_signer: true,       // Must sign
    is_writable: false
  }
]
Why Mark This? Sealevel uses it for parallel processing. Read-only = parallel safe. Writable = must be sequential.

Fee Payer

Transaction includes:
├─ Payer: First signer
├─ Fee: 5000 lamports (~0.000005 SOL)
├─ Deducted from: Payer's lamports
└─ If insufficient: Transaction fails

Who pays?
├─ Usually: Transaction creator
├─ Can be: Different (subsidized TX)
└─ Example: Company pays for employee TXs

Signatures Using Ed25519

Signing Process:
│
├─ Message (serialized to bytes)
│  ├─ Account addresses
│  ├─ Recent blockhash
│  └─ Instructions
│
├─ Per Signer:
│  ├─ Private key + Message → Ed25519 Sign
│  └─ Get 64-byte signature
│
└─ Add all signatures to transaction

Multiple Signatures (Multisig):

Example: 3-of-3 multisig approval

Transaction: "Transfer treasury funds"

Signatures:
├─ Alice signs (proves authorization)
├─ Bob signs (proves approval)
└─ Carol signs (proves approval)

Verification:
├─ Check all 3 signatures valid
├─ All valid = transaction proceeds
└─ Any invalid = transaction fails

⚡ Transaction Lifecycle

Step-by-Step Flow

Step 1: Create & Sign ├─ Fetch recent blockhash ├─ Create message └─ Sign with private key Step 2: Submit to Network ├─ Send to RPC node ├─ Validate format/signatures └─ Enter mempool Step 3: Leader Includes Block ├─ Leader pulls from mempool ├─ Orders transactions ├─ Bundles into block Step 4: Execute (Sealevel) ├─ Load accounts ├─ Run program logic ├─ Modify states atomically └─ Revert all if error Step 5: Broadcast (Turbine) ├─ Break into pieces ├─ Spread through network └─ Validators receive Step 6: Validators Vote ├─ Check signatures valid ├─ Vote on block (Tower BFT) └─ Votes accumulate Step 7: Finality (~6.4 sec) ├─ 2/3+ of stake votes ├─ FINAL (cannot revert) └─ ✓ Success

State Changes

Before TX:
├─ Account A: 1000 lamports, 100 tokens
└─ Account B: 2000 lamports, 50 tokens

TX: Transfer 20 tokens A→B

After TX (Committed):
├─ Account A: 995 lamports, 80 tokens (20 sent + 5000 fee)
└─ Account B: 2000 lamports, 70 tokens (received 20)
✓ Atomicity: Either ALL changes happen or NONE. Fee always deducted even if TX fails.

🛡️ Replay Protection

The Replay Attack

Scenario:
1. Alice sends "Transfer 10 SOL to Bob" (signed)
2. Attacker hears transaction
3. Attacker rebroadcasts 1000 times
4. Bob gets 10,000 SOL instead of 10!

Solution 1: Recent Blockhash (Default)

Include in TX:
├─ Recent blockhash (hash of block 100 blocks ago)

Validator checks:
├─ Is this blockhash still valid?
├─ Default: Yes, if within 2-3 minutes
├─ Too old: REJECT (replay!)
└─ Not recognized: REJECT (different chain?)
Time Block Hash TX Valid?
Slot 1000 0xAABBCC... ✓ Created now
Slots 1001-1099 Other hashes ✓ Still valid
Slot 1100 0xAABBCC no longer valid ✗ Too old
Slot 1101+ - ✗ Outside window
Why This Works: Blockhash expires after 2-3 minutes. Attacker can't replay after that. Must use current blockhash, making it a new transaction.

Method 2: Durable Nonce (Advanced)

The Problem

User signs TX at time T
TX might submit at time T + 5 minutes
Recent blockhash now expired!
TX fails :(

Solution: Nonce Account

Nonce Account Structure:
├─ nonce_value (32 bytes, changes each use)
├─ authority (who can use this)
└─ fee_calculator

TX Builder:
├─ Instead of recent_blockhash
├─ Use nonce_value from Nonce Account
├─ Can submit anytime!

Each Use:
├─ Nonce advances to new value
├─ Old nonce forever invalid
├─ Prevents replay forever
Recent Blockhash
2-3 min
Durable Nonce
∞ (until used)

Who Creates Nonce?

User creates it:
├─ Pay rent-exempt balance (~5000 lamports)
├─ One-time cost
├─ User is owner
└─ Can be reused many times

Cost Analysis:
├─ Recent blockhash: Free
├─ Durable nonce: 5000 lamports once
└─ Worth it for: Delayed/scheduled TX

🎯 Validators and Authority

Validator Architecture

Validator isn't ONE account, it's SEVERAL:

1. VALIDATOR IDENTITY
   ├─ Purpose: Identify validator
   ├─ Used for: Signing blocks
   └─ Risk: High (if leaked = attacks possible)

2. VOTE ACCOUNT
   ├─ Purpose: Store voting records
   ├─ Owner: Vote Program
   └─ Update: When voting

3. STAKE ACCOUNT
   ├─ Purpose: Hold staked SOL
   ├─ Owner: Stake Program
   └─ Associated: With validator

4. COMMISSION ACCOUNT
   ├─ Purpose: Collect fees
   ├─ Owner: Validator
   └─ Receives: Validator's share

Authority Types (Different Keys!)

Authority Type Function Risk Level
Validator Identity Keypair Signs blocks CRITICAL
Vote Authority Address Modify vote account High
Stake Authority Address Add/remove stake Medium
Withdraw Authority Address Withdraw SOL Medium
⚡ Can Be Different People! Corporate validator can have operator (validator identity) + CFO (withdraw authority) + board (vote authority).

Vote Account

Vote Account Data:
├─ node_pubkey (validator ID)
├─ authorized_voters (list of who can vote)
├─ authorized_withdrawer (who can withdraw)
├─ commission (0-100%)
├─ votes (last 40-50 recent votes)
├─ root_slot (most recent finalized)
└─ prior_voters (history)

Who Votes? Validator's software signs blocks, calls Vote Program, updates vote account.

Stake Account

Stake Account Data:
├─ delegation
│  ├─ voter_pubkey (validator's vote account)
│  ├─ stake (amount staked)
│  └─ activation_epoch (when started)
├─ authorized
│  ├─ staker (can add/remove stake)
│  └─ withdrawer (can withdraw)
├─ lockup
│  ├─ custodian (can change during lockup)
│  ├─ epoch (when lockup ends)
│  └─ unix_timestamp (when lockup ends)
└─ rent_exempt_reserve (minimum balance)

Staking Process

Step 1: Create Stake Account
├─ Allocate new account
└─ Fund with SOL

Step 2: Activate Stake
├─ Specify validator
└─ Activation epoch = current + delay

Step 3: Earn Rewards
├─ Every epoch: rewards calculated
├─ Based on: validator performance
└─ Typically: 3-8% annually

Step 4: Inactivate
├─ Inactivation epoch = current + 1
└─ Can't withdraw until then

Step 5: Withdraw
├─ Remove SOL to preferred account
└─ Requires: withdrawer authority

Commission Example

You have 100 SOL staked
Validator commission: 5%
Validator earns 10% yield

Your rewards:
100 × 10% × 0.95 = 9.5 SOL

Validator's commission:
100 × 10% × 0.05 = 0.5 SOL

Leader Schedule

What: Predetermined list of which validator leads each slot.

Example (simplified):
Slot 1000: Validator A leads
Slot 1001: Validator B leads
Slot 1002: Validator C leads
Slot 1003: Validator A leads
...
(432,000 slots per epoch)

How It's Determined

Before each epoch:
├─ List all active validators
├─ Sort by stake (more stake = more slots)
├─ Use pseudorandom algorithm
├─ Result: deterministic but unpredictable

Probability:
Validator i = (stake_i / total_stake) × 100%

Example:
├─ Validator A: 100k SOL = 57% slots
├─ Validator B: 50k SOL = 29% slots
└─ Validator C: 25k SOL = 14% slots
✓ Benefits: Everyone knows schedule in advance. Prevents "leader bribery". Validators can prepare blocks early via Gulf Stream.
