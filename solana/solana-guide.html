🌍 What is Solana?

Layman's Explanation

Imagine a bank that processes transactions for millions of people. The bank needs to keep track of who has how much money, verify that transactions are real, and make sure nobody cheats the system.

Solana is a global, decentralized bank that lives on computers all over the world instead of in one building. Nobody owns it, but thousands of computers (called nodes) work together to:
  • Keep track of who owns what
  • Verify that transactions are real
  • Process transactions very quickly
  • Prevent fraud

Think of Solana as a super-fast, transparent ledger that everyone can see and verify, but nobody can control or fake.

Technical Explanation

Solana is a Layer 1 blockchain using cryptographic proof and consensus mechanisms to validate transactions without central authority.

❓ Why Was Solana Created?

The Problem (2019-2020)

Bitcoin and Ethereum were slow and expensive:

Bitcoin
~7 tps
Ethereum
~15 tps
Visa
~65,000 tps

The Solana Solution

Anatoly Yakovenko invented Proof of History enabling:

  • Order transactions using time instead of consensus
  • Process thousands of transactions in parallel
  • Achieve massive throughput while staying secure
  • Keep transaction fees extremely low (~$0.00000325)

⏰ Proof of History

The Problem

How do validators agree on the ORDER of transactions without asking each other?

The Solution

Proof of History proves what happened WHEN without asking anyone else!

Like a Tamper-Proof Journal

  1. Alice writes: "I bought coffee for $5"
  2. She puts a unique seal on it
  3. Bob writes next and references Alice's seal
  4. Bob's entry gets a new unique seal
  5. Eva writes next and references Bob's seal
Result: The ORDER is mathematically proven. No voting needed. No delays.

💰 Proof of Stake

Why Not Proof of Work?

Proof of Work (Bitcoin): Miners solve hard puzzles. Uses massive electricity. Secure but slow and expensive.

Proof of Stake (Solana): Validators put up collateral. Honest behavior = rewards. Dishonest = lose stake.

🏦 Escrow Analogy: Like a security deposit—if you have $1M staked and lie, you lose it. Cheaper to be honest!

How It Works

  1. Setup: Validators deposit SOL
  2. Behavior: Validators follow rules and participate
  3. Rewards: Honest validators earn fees + rewards
  4. Punishment: Dishonest validators lose stake

🏗️ Solana Architecture

The Solana Stack

Applications (dApps, wallets, games)
                    ↓
Users submit transactions
                    ↓
         Solana Network (validators)
                    ↓
    ┌────────┬──────────┬────────┬────────┐
    ↓        ↓          ↓        ↓
 Gulf    Turbine  Sealevel  Tower BFT
Stream (broadcast) (parallel) (consensus)

Gulf Stream (Mempool)

Forward transactions to next leaders early. By the time they become leader, your transaction is ready!

Turbine (Broadcast)

Break blocks into pieces and spread through network layers. Like Netflix streaming video.

Sealevel (Runtime)

Process transactions in parallel. Multiple workers handling different transactions simultaneously instead of one-by-one.

Tower BFT (Consensus)

Voting system for validators to agree on blocks. Network with up to 33% bad actors still reaches agreement when 2/3+ vote.

✅ Key Takeaways

  • Solana is a high-speed, low-cost blockchain
  • Invented to fix Bitcoin/Ethereum's speed limits
  • Uses Proof of History + Proof of Stake + parallel processing
  • Processes 400-1000 tps vs 7-15 for Bitcoin/Ethereum
  • Average transaction cost is $0.000003
  • Architecture: Gulf Stream → Turbine → Sealevel → Tower BFT

🔐 Cryptography Fundamentals

Public Key Cryptography

Simple Analogy:

  • You have a private key (secret key to your lock)
  • You give others your public key (the lock itself)
  • Anyone can lock using your public key
  • Only you can unlock with private key

💾 Account Model

Account Structure

Account:
├─ address (32 bytes)
├─ owner (32 bytes, who controls)
├─ lamports (balance)
├─ data (raw bytes)
├─ executable (is this a program?)
└─ rent_epoch (tracking)

Lamports

1 SOL = 1,000,000,000 lamports
1 lamport = 0.000000001 SOL

Rent System

Annual Rent = (account_size) × 19.1 lamports/byte

Rent Exemption:
Balance ≥ 2 years rent = never pay rent

Associated Token Accounts (ATA)

Derive deterministic address from: owner + token_mint

Benefit:
├─ Don't need to ask for address
├─ Standard location per token
└─ Auto-create if missing

Program Derived Addresses (PDA)

No private key! Derived from program ID + seeds + bump.

Use Case: Program vault holding funds
├─ Only program controls it
└─ No private key to steal

Fee Payer

Transaction:
├─ Payer: First signer
├─ Fee: 5000 lamports (~$0.000005)
├─ Deducted from: Payer's lamports
└─ If insufficient: Transaction fails

Signatures (Ed25519)

Process:
├─ Serialize message
├─ Sign with private key
└─ Get 64-byte signature

⚡ Transaction Lifecycle

1. Create & Sign
   ├─ Fetch blockhash
   ├─ Create message
   └─ Sign

2. Submit to Network
   ├─ Send to RPC
   ├─ Validate
   └─ Enter mempool

3. Leader Includes
   ├─ Pull from mempool
   ├─ Order
   └─ Bundle into block

4. Execute (Sealevel)
   ├─ Load accounts
   ├─ Run program
   └─ Modify states

5. Broadcast (Turbine)
   ├─ Break into pieces
   └─ Spread

6. Validators Vote
   ├─ Check signatures
   ├─ Vote (Tower BFT)
   └─ Votes accumulate

7. Finality (~6.4 sec)
   ├─ 2/3+ of stake votes
   └─ ✓ FINAL

🛡️ Replay Protection

The Replay Attack

Attacker rebroadcasts transaction 1000 times = duplicate execution

Method 1: Recent Blockhash (Default)

Include in TX:
├─ Recent blockhash
├─ Validator checks: Still valid?
├─ Default: Yes, if within 2-3 minutes
└─ Too old: REJECT (replay!)

Method 2: Durable Nonce (Advanced)

Nonce Account:
├─ nonce_value (32 bytes, changes)
├─ authority (who uses)
└─ cost: ~5000 lamports once

Benefits:
├─ Can submit anytime (not time-limited)
├─ Nonce advances with each use
└─ Old nonce forever invalid
Method Valid For Cost Use Case
Recent Blockhash 2-3 min Free Normal TX
Durable Nonce Forever 5k lamports Delayed TX

🎯 Validators & Authority

Validator Architecture

1. VALIDATOR IDENTITY
   ├─ Purpose: Identify validator
   └─ Used for: Signing blocks

2. VOTE ACCOUNT
   ├─ Purpose: Voting records
   ├─ Owner: Vote Program
   └─ Update: When voting

3. STAKE ACCOUNT
   ├─ Purpose: Hold staked SOL
   └─ Owner: Stake Program

4. COMMISSION ACCOUNT
   ├─ Purpose: Collect fees
   └─ Owner: Validator

Authority Types

Authority Function Risk
Validator Identity Signs blocks CRITICAL
Vote Authority Modify vote account High
Stake Authority Add/remove stake Medium
Withdraw Authority Withdraw SOL Medium

Leader Schedule

Predetermined list of leaders per slot:

Slot 1000: Validator A
Slot 1001: Validator B
Slot 1002: Validator C
...

Probability:
Validator i = (stake_i / total_stake) × 100%

Benefits:
├─ Schedule known in advance
├─ Prevents leader bribery
└─ Validators prepare early

Staking Process

1. Create Stake Account
2. Activate (specify validator)
3. Earn Rewards (3-8% annually)
4. Inactivate (when done)
5. Withdraw SOL
